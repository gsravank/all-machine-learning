Detect Fake Profiles

Problem: In recent years fake profiles has been an issue that we need to deal with on our website. Can you design a system where we can detect fake profiles?

Clarifying Requirements

First, you should clarify the requirements and we have written about how to do this in the Clarifying Requirements section of the ML Design Template.

Metrics

Offline: 𝑃𝑟𝑒𝑐𝑖𝑠𝑖𝑜𝑛=𝑇𝑃𝑇𝑃+𝐹𝑃, 𝑅𝑒𝑐𝑎𝑙𝑙=𝑇𝑃𝑇𝑃+𝐹𝑁 and 𝐹1=2∗Precision∗RecallPrecision+Recall==2∗TP2∗TP+FP+FN where

  • True Positive (TP): Correctly detected a fake profile when the profile was fake
  • True Negative (TN): Correctly predicted the profile is real when the profile was indeed real
  • False Positive (FP): Incorrectly detected a fake profile when the profile was real
  • False Negative (FN): Incorrectly predicted the profile is real when the profile was fake

Online: A good online metric would be the number of times a user whose profile was deemed fake was appealed successfully. That would mean that our detection was incorrect, so we would hope this number to be low.

Architecture

We would detect fake profiles at two times:

  1. At Registration – this would be a classification model that determines if the user is a fake profile or not
  2. After registration – this would be monitoring profiles and how they interact with the platform and other users. This is a three-step process (Xiao, Cao & Freeman, David & Hwa, Theodore. (2015). Detecting Clusters of Fake Accounts in Online Social Networks. 91–101. 10.1145/2808769.2808779)
    1. Cluster Builder – Given several accounts, we group them into clusters (hyper-parameters cluster size, account registration, clustering criteria – registration IP address).
    2. Profile Featurizer – Converts the data from each cluster into a single vector representation which we will pass into the next step for machine learning. Some of the features used in building this cluster would be mean or quartiles for numerical features, number of unique values for categorical features, patterns in email addresses and if the frequency of names in a cluster is mainly from top-ranked first and last names.
    3. Account Scorer – We create a model to determine the likelihood of a cluster being composed of fake accounts. Once we have the probability score we can then automatically ban the accounts, send the accounts for manual review or the score is low indicating that there are no fake accounts so we do nothing.

Data

This is section and the next would be where you would probably spend most of the time in the interview and where the interviewer would be looking to see how you perform.

  1. First, identify the target variable and how you would collect and label it (if necessary)
    • We would use the data of previously banned fake accounts as our target variable
  2. Discuss features and possible feature crosses
    • At registration: email, phone number, IP address, name, OS Version, MAC, address, hardware, VPN
    • Over time: number of interactions with other users, number of friends, number of posts, frequency of posts/comments, number of times they have been reported by other users, average clicks per session, average session length, the content of messages.
  3. Data preprocessing/ Feature Engineering
    • We will have text, numerical and categorical features and so we would need to preprocess them. Please refer to the Data section of the system design template.
    • Train-test split: 70/30 split with more recent datapoints in the test set
    • Another important point is the balancing positive and negative training examples: For our case, the number of positive examples (users who have fake profiles) is significantly less than the negative examples (normal users) so we can use a number of techniques to solve this problem (see them in the the Data section)
  4. Feature Selection
    • When we are using a deep neural network, we do not need feature selection. For other methods we can use tree-based estimators can be used to compute feature importances. Additionally, we can use the L1 norm for regularization so that some feature coefficients would be zero.
  5. Additional Considerations
    • Biases: We need to ensure we sampling from a large enough subset of demographics, if not maybe we can group the smaller demographics together

Once we have completed our discussion about data you should ask the interviewer if they would like you to explain how to productionize this data step.

​Model

Since detecting fake users at registration and creating an account scorer are both classification problems. We can use typical models to solve them:

  • Logistic Regression
  • XGBoost
  • NN Model

For the Cluster Builder we can use some popular clustering algorithms:

  • K-means: will find clusters by minimizing the variance of data points within each cluster
    • Pros: it is simple to implement, guarantees that there will be conversion
    • Cons: k has to be chosen manually, outliers affect clusters, does not scale to the number of dimensions.
    • Loss: ∑𝑖∑𝑘𝐼𝑖𝑘‖𝑥𝑖–𝜇𝑘‖2 where 𝑥𝑖 is the 𝑖-th datapoint, 𝜇𝑘 is the center of cluster 𝑘 and 𝐼 is the indicator function (1 or 0) if 𝑥𝑖 belongs to cluster 𝑘.
  • DBSCAN: finds different clusters based on the density of regions within the dataset.
    • Pros: do not need to specify the number of clusters, robust to outliers, build clusters of arbitrary shape
    • Cons: if the dataset has varying densities of clusters then the model would not work good, you still need to specify some hyper-parameters.
  • Gaussian Mixture Model: will use multiple Gaussian distributions to find clusters in the fake profile data.
    • Pros: data points can be in more than one cluster, clusters can be of arbitrary shape
    • Cons: more difficult to interpret because data points can belong to multiple clusters, it usually takes longer to run than the other methods
    • Loss: Expectation–maximization algorithm

Serving

Now you have to serve the model to users, so there should be a few talking points:

  1. Online A/B testing:
    • Online Metric: the number of times a users’ ban was appealed successfully
    • Sample size: we can use 𝛼=0.05 and power=0.8 to perform this calculation
    • We will split users randomly via each search session
  2. For detecting fake users at registration will our model online and this we need to be aware of scalability or latency issues (load balancing), we may need to create a feature store to lookup features at serve time and maybe some caching depending on the use case. For detecting fake users after registration, we can run an offline model which might run every day or every week depending on how much fake users affect our website.
  3. We can monitor our performance with a few metrics like if there is any performance drop, the latency of our system and if there are any biases in the model.
  4. Where to run inference: in this case we would want to run inference on our server side because we do not want to wait for users to update their app or similar to get our updated models detection.
  5. We will need a re-training strategy that can keep up with bots and adversarial actors. Thus, we would want to update our model at least once a month, however, the more often we can do this the better.
